# Master Brainstorming Document – Migration & Backward Compatibility Adapter (FEAT-093)

## 1. Executive Summary
This document designs the **Migration & Backward Compatibility Adapter**. It provides compatibility interfaces to translate between legacy procedural structures and the new goal-driven runtime architecture, ensuring that existing skills, state configurations, and Visualizer extensions continue to function without modifications.

## 2. Background
Transitioning AIWF to a goal-driven operating system requires major changes to state databases, state schemas, and CLI commands. However, the system must remain backward-compatible to prevent breaking existing developer workflows, current visualizer sidebar integrations, and legacy CLI scripts.

## 3. Current Architecture Analysis
The current codebase has `migrate_session_to_db.py` under the `.agents` folder, indicating a need for database schema migrations.
- State file formats: Flat JSON files containing checkpoint numbers and file lists.
- Existing CLI commands are parsed inside `workflow_runtime.py`.

## 4. Current Limitations
- **Fragile State Schema**: Direct modifications to JSON schemas can break the Visualizer.
- **Incompatible Commands**: Legacy skills expect standard sequential commands, but the new loop is goal-driven.
- **Data Migration Gaps**: Updating active workspaces can corrupt historical telemetry records.

## 5. Objectives
- Develop a **Telemetry Schema Adapter** to format new event streams into legacy shapes.
- Implement a **CLI Command Compatibility Wrapper** to translate old actions.
- Provide **Automated Migrations** for state databases.

## 6. Functional Requirements
- **FR-01: Legacy State Adapter**: Intercept state queries from Visualizer; return mapped legacy schemas.
- **FR-02: Command Translation Wrapper**: Map commands like `start` or `complete` to appropriate goal-loop iterations.
- **FR-03: Telemetry Database Migration**: Automatically run schema migrations on the SQLite database during initialization.
- **FR-04: Skill Execution Bridge**: Wrap existing v2/v3 skills so they can run inside the new task-graph loop without changes to their instructions.

## 7. Non-Functional Requirements
- **NFR-01: Performance**: Translation overhead for CLI wrapper must be `< 5ms`.
- **NFR-02: Zero Manual Setup**: Database migrations must run automatically on startup without user intervention.

## 8. Scope
- State conversion wrappers (`state_adapter.py`).
- CLI command mapper hooks.
- SQLite migration runner scripts.
- Skill adapter wrappers.

## 9. Out of Scope
- Supporting deprecated python/node environment versions.
- Visualizer source code modifications (restricted to output adapters).

## 10. Runtime Responsibilities
The Compatibility Adapter intercepts CLI command calls, maps payload fields to legacy schemas, manages database migration versions, and logs translation warnings.

## 11. Components
- `StateAdapterEngine`: Maps event logs and tree objects to legacy state JSON files.
- `CLICommandWrapper`: Rewrites incoming legacy commands before dispatching to the loop.
- `DatabaseMigrator`: Executes SQL migration scripts.

## 12. Data Model
```
[Legacy Visualizer Query] ──> [StateAdapterEngine] ──(reads new context)──> [Maps schemas] ──> [Returns legacy shape]
```

## 13. Runtime State
```
[Unmigrated DB] ──(run migrator)──> [Migrating] ──(success)──> [Compatible Run]
```

## 14. Event Flow
1. Visualizer queries `.agents/state/context.json` to render the UI.
2. The runtime uses the StateAdapterEngine to intercept the read.
3. The engine parses the active Goal Tree, converts it into a checkpoint index (1-9), and generates the legacy schema.
4. The Visualizer renders the UI without errors.

## 15. Sequence Flow
- Query -> intercept -> map fields -> format schema -> return response.

## 16. Dependencies
- Executive Runtime (FEAT-086) and Event Journal (FEAT-089).

## 17. Integration Points
- CLI: `python workflow_runtime.py migrate`
- Database: `.agents/project_runtime.db` and legacy `workflow_usage.db`

## 18. Interaction with Executive Runtime
- The Executive Loop calls the Migration Runner during startup checks to verify database versions before execution.

## 19. Interaction with other features
- Translates legacy status updates generated by older **Skills (FEAT-009)**.

## 20. Security Considerations
- Validate session values before running SQL migrations to prevent DB injection.
- Backup the telemetry database before executing any write migrations.

## 21. Performance Considerations
- Run database migrations in a separate transaction; roll back immediately on failure.

## 22. Scalability Considerations
- Modular migration files allow adding new version migration scripts easily.

## 23. Failure Scenarios
- **DB Migration Fails**: Print error to stderr, roll back transaction, copy backup, and start in read-only compatibility mode.
- **Unsupported Legacy Skill Called**: Warn user and run via compatibility wrapper.

## 24. Recovery Strategy
Maintain copy backups of the SQLite databases inside `.agents/runtime/backups/`. If corruption occurs, restore the backup copy and restart the migration process.

## 25. Migration Strategy
Run a check on initialization. If the DB contains legacy tables, backup files, execute the migration script block, update schema version, and log output.

## 26. Backward Compatibility
Legacy CLI commands like `step` and `complete` continue to run successfully by translating their parameters internally into task graph state updates.

## 27. Risks
- Data loss during telemetry database migrations. Mitigated by enforcing automatic database backups.

## 28. Alternative Designs
- **Option A**: Break compatibility. (Rejected: ruins developer experience; forces upgrading all plugins).
- **Option B**: Maintain two separate databases. (Rejected: increases sync complexity and disk bloat).

## 29. Trade-offs
- Maintaining compatibility code adds wrapper layers, but prevents development workflow interruptions.

## 30. Acceptance Criteria
- [ ] AC-01: Successfully execute legacy CLI commands without errors.
- [ ] AC-02: Run DB migrations cleanly without losing historical telemetry logs.

## 31. Estimated Complexity
- Medium.

## 32. Blueprint Estimation
- 1 design blueprint (`docs/designs/FEAT-093_compatibility_migration_adapter.md`).

## 33. Recommended Implementation Order
Implement last, as it wraps all other modules to guarantee compatibility.
